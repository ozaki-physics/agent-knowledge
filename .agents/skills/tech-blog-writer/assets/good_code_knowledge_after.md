## 心得の前提
じゃあ 相手の要求を 長期的に たくさん実現するには どうしたらよいでしょうか?  
それは 相手の要求や求める機能 を __早く__ かつ __確実に動作して__ 相手に届けることです。  

じゃあ 早く かつ 確実に動作して 相手に届けるためには どうしたらよいでしょうか?  
1. 早く 提供するには __コードが読みやすく機能追加や変更が簡単__ な状態を維持すればよいです。  
2. 確実に動作して 提供するには __変更してもバグらず安定__ な状態を維持すればよいです。  

つまり __内部品質 を上げる__ ことで 相手の要求をたくさん実現できます。  
内部品質 を上げる方法は 常に勉強しましょう。  
我々は IT業界に就職して プログラミングで給料をもらっているプロなので __勉強は一生していく必要があります__  
### コードの保守性や可読性が上がる副次的なメリット  
- 開発しやすくなり 生産性が上がる  
- コードレビューが簡単になる  
- 新規参入者や若手でも キャッチアップが容易になる  
- 保守性や可読性が考えられるエンジニアにレベルアップする  

## コードにおける前提
### __コードは必ず変更される__  
変更する未来が来る なら __コードは 変更しやすく書く__ 必要があります。  
### __コードは 書くために使った時間 より 読まれる時間の方が長い__  
読まれる時間が長い なら __コードは 読みやすく書く__ 必要があります。  

## 保守性, 可読性 を上げるための 心得
### 1. コードが最重要ドキュメント 妥協して書くな!  
__ユーザーに影響を与えるのは コード__ です。  
仕様書や設計書は あくまで コードを作るための過程で必要な、 認識合わせの資料です。  
仕様書や設計書は 不具合のときの 証拠の意味もありますが 仕様だからと正当化しても ユーザーには 関係ありません。  
ユーザーにとって 意図した動きをしていないなら そのソフトウェアは使い物にならず ゴミです。  
よって ユーザーに影響を与えるのは コード です。  
__ユーザーへの価値提供が一番大事なので コードが一番重要__ なのです。  

伝えたいことは、 プログラマー は __一番重要なものを作っている意識を持ちなさい__ ということです。  
まぁコレくらい いいか という 妥協 や いい加減さ は、 <a href="https://www.kuins.ac.jp/news/2019/02/post_20.html" target="_blank" rel="noopener noreferrer">割れ窓理論</a> のように、 いつか 大きなトラブルに発展します。  
コードはしっかり作りましょう。  

### 2. 動いてないコードは 消せ!  
コードの可読性を上げるため __コードのノイズを消す__ こと。  

ノイズ(コメントアウトされた過去のコード など) について 以下の運用にする。  
目的: コードは 常にノイズ(コメントアウトされた過去のコード)が無い状態にする。  
ポイントは、 確認者が コードの差分を認識しやすくすること。  
整えた結果(ノイズの削除やコードフォーマットを通したコード) と 行った改修 は __別コミットにする__ こと。  

1. 自分が編集したいファイルやメソッドで コメントアウトされた過去のコード があった場合、 コメントアウトされた過去のコード を 消して 一度コミットする  
消す範囲は ファイル全体, 自分と関わるメソッド, メソッドの中の一部 のいずれでもよい  
2. 自分の改修を行う。 このときは ADD/DELL を書いて コミットする (改修前のコードはコメントアウトされた状態)  
3. コードレビュー や IT(結合テスト)で 確認者にチェックしてもらう  
4. OK がもらえたら 自分が改修した部分の コメントアウトされたコード を 消して コミットする  

ここでは ノイズとして コメントアウトされた過去のコード を例に挙げているが 他にも ノイズと判断できるものは 消して良い  
例えば 誰が改修したかの履歴の文字列(ADD/DELL), メソッド名にある Docs のヒストリー  

ただし クラス名の直下に書かれている ヒストリー は 直近2年分は消してはいけない  
毎年 どれだけ改修したかの記録文書を作るときに利用するため  
(直近1年度分だけでもいいかもしれないが、 記録文書作成の区切りが分からないため 直近2年にする)  

### 3. 命名をサボるな!  
名前が分かりやすいと どんな意図で何をしているか分かりやすくなり、 可読性が上がります。  
変数名, メソッド名, クラス名, モジュール名, ファイル名 など 名前を付けるタイミングは すべて意識してください。  
また 適切な名前が付けられない ということは、 自分が何をしたいか分かっていない と判断できる指標にもなります。  
よって __名前はサボらず丁寧に付けなさい__。  

### 4. コメント は 意味や意図が伝わるように書け!  
__コメントは 意味や意図を伝えるために書きなさい。__  
コードで行っていることを日本語にしただけのような 意味のないコメントは ノイズになるので止めなさい。  

<details>
    <summary>例</summary>
    <div class="details-content-indent">
      <div>'' 計算して再代入する</div>
      <div>price = price * 1.1</div>
      <div>これは コードを読めば分かることで 意味がないコメントになっているので止める</div>
      <div>↓</div>
      <div>'' 消費税を加える</div>
      <div>Dim totalPrice = price * 1.1</div>
    </div>

### 5. ネスト は 減らせ!  
if 文, for 文, メソッド呼び出し など、 ネスト(入れ子)が深いコードは とても読みにくいです。  
通るための条件や状態を 脳は覚えていられず 読みにくい = 変更しにくい となり 保守性が下がります。  
__ネストは減らしなさい__。  

深いネスト になったら 以下のような ネストを減らす工夫を考えてください。  
- 条件が多いなら 別のメソッドに切り出せないか?  
- 早期リターン(ガード句)を活用できないか?  

### 6. 変数 は 使い回すな!  
変数は __その時点で 何を表しているのか__ が重要です。  
途中で 変数の役割が変わると この変数は どの時点で どんな値を持っているのか? を常に意識させられ、 バグの原因にもなります。  
__変数は 分かりやすい名前で 使い回さず 定義しなさい__。  
ただし 既存の使いまわしている変数を わざわざ変えなくていいです。(例えば DataSet の ds など)  
自分が新しくコードを書くときは 意識してください。  

### 7. メソッド は 小さな粒度 で 作れ!  
メソッド は __小さい粒度 かつ 単一の責任を持たせる__ ことが 理想です。  
(処理内容にもよるが 理想は 30行程度, 多くても 50行, 100行を越えたら異常)  
原則として __1つのメソッドで 1つのことを行う__ という 単一責任の原則 を意識しながら、 メソッドを分割しましょう。  
複数のことを 一度にやろうとすると、 メソッドが肥大化し、 何をしているメソッドなのか 分からなくなります。  
何をしているか分からないメソッド は バグりやすく 保守性が下がる一方です。  

### 8. 設計書のキーワードをコードに書け  
設計書に存在する キーワードになりそうな単語を コードに含めること。  
なぜなら 設計書のどの部分 が コードのどの部分 に対応するか分かりやすくするため。  
現在は キーワードを入れてコードと対応させる必要があるぐらい コードが読みにくい。  
将来的には コードの可読性が十分に高くして このルールを削除する
