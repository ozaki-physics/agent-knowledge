## 目的  
保守性の高いコード を書くために 抑えるポイントを学ぶ  
コードレベル での 基礎 と 応用 部分を知る  
クラス設計 を勉強する  

## 保守性の高さを表す指標  
保守性の高いコード とは 以下の3つを満たすコードです。  
- 凝集度 が __高い__  
- 結合度 が __低い__  
- 循環的複雑度 が __低い__  

### 凝集度  
凝集度 とは モジュール内の __データとロジックの関係性の強さを表す指標__ です。  
モジュールとは 任意のまとまり(クラス, パッケージ, レイヤー など)を表します。  
ざっくり 言うなら モジュールのまとまり具合に関する指標 です。  

__凝集度 が 高い__ ほど 変更容易性 が 高く __保守性の高いコード__ になります。  
凝集度 を表す 指数 として LCOM(Lack of Cohesion of Methods) があり 計測ツール もあります。  
### 結合度  
結合度とは モジュール間の __依存の度合いを表す指標__ です。  
ざっくり言うなら 適切に分割できているか示す指標 です  

__結合度 が 低い__ ほど 変更容易性 が 高く __保守性の高いコード__ になります。  

具体的には 以下の状態が 結合度が高い 良くない状態です。  
- 他の モジュール の 関数 を 直接 呼んでいる  
- 1個 の ファイル で import 文 が 複数 ある  
- グローバル変数 を たくさん 参照している  
- メソッド の 引数 が 多い  
- ファイル や データベース に 頻繁に アクセス している  

結合度 を表す 指数 として Coupling があり 計測ツール もあります。  
ただ 計測ツール を使わなくても 呼び出してるクラスが多いと 結合度が高いと予想できます。  

__結合度が高いと__ 1個のメソッドで 多くのことを処理しており __単一責任 でない__ = 保守性 が 低い です。  

### 循環的複雑度  
主にメソッドを対象に __コードの構造的な複雑さを示す指標__  
別名: 循環的参照度, サイクロマティック複雑度  

条件分岐 や ループ処理 が増える, ネストがある と 複雑さは増えます  
その複雑さの度合いを 計測する数値 です。  

計測するツールもあります  
指数のイメージとしては 以下です  
- 1関数あたり初期値が1  
- 分岐コード (If, for, case, while など)ごとに1加算  
- 直線で進むものは カウントしない(else, default など)  

目安として 以下と言われています。  
ドキュメント: <a href="https://jp.mathworks.com/discovery/cyclomatic-complexity.html" target="_blank" rel="noopener noreferrer">循環的複雑度の目安</a>  
| 循環的複雑度 | 複雑さの状態 | バグ混入確率 |  
| - | - | - |  
| 10以下  | 非常に良い構造 | 25% |  
| 30以上 | 構造的なリスクあり | 40% |  
| 50以上 | テスト不可能 | 70% |  
| __75以上__ | __いかなる変更も誤修正を生む__ | 98%  

```java  
void function()  
{  
    if(条件式)  
    {  
        処理 ;  
    }  
}  
```

## クラス単体で動作する  
今からは 上記の概念のうち 凝集度 を 中心に クラス設計 について 話していきます。  
__凝集度が高い クラス__ とは __インスタンス変数 と そのインスタンス変数を使うロジック が 同じクラス内にある__ 状態です。  
つまり クラス単体で動作する とも言えます。  

クラス単体で動作するとは 例として 以下を満たすものです。  
- そのクラスを使うために 初期設定 が 必要ない  
- そのクラスを 正しく操作できるメソッドのみ 外部に公開している  
- メソッドは 必ず インスタンス変数 を使っている  

<details>  
    <summary>現実でのイメージ</summary>  
    <div class="details-content-indent">  
      <div>イメージとしては キーボード や マウス は パソコン に接続するだけで 使えます。</div>  
      <div>キーボードのボタンを付けたり マウスを分解して 他の部品を付けてからじゃないと まともに使えない ということはありません。</div>  
      <div>初期設定が必要ないですよね</div>  
      <div>また キーボード自体が 光ったり 明るさ調整ができたりします。</div>  
      <div>その明るさ調整をミスると キーボードが壊れる ということもありません。</div>  
      <div>正しく操作できるメソッドのみ 外部に公開していますね</div>  
    </div>  
</details>  

逆に 単体で安全に使えないクラスは 使う気が失せます。  
なぜなら そのクラスが バグってないか 気をつけながら 使うので 神経がすり減ります。  
また 気をつける事が多いので 開発が遅くなります。  
例えば 以下のような状態です。  
- 他のクラス や メソッド で 初期化 が 必要  
- 毎回 インスタンスから取れるデータ が 正しいのか チェックしてから使う  

ソフトウェア の 部品である __クラス 1個1個 が 安全 に 使えれば__ ソフトウェア全体の品質も上がり __保守性の高いコード__ になります。  

### コンストラクタで 確実に 正常値を設定する  
昔は コンストラクタ は引数が無く インスタンス を ただ 生成するだけでした。  
そして その後に セッター で値を入れていました。  
```java  
// Money クラス  
Class Money {  
  int 金額;  

  // コンストラクタ  
  public Money(){  
    // 何もしない  
  }  

  // セッター  
  public void set金額(int 入力){  
    this.金額 = 入力  
  }  
  // ゲッター  
  public int get金額(){  
    return 金額;  
  }  
}  

// 用意するとき  
Money money = new Money()  
money.set金額(100)  

// 使うとき  
print(money.get金額())  
```  

でも これは 問題があります。  
もし 用意するときに __セッター を忘れたら__ 使うときに null で __エラー__ になります。  

じゃあ 絶対に値を格納するには どうしたらいいでしょう?  
それは __コンストラクタ で 初期化をする__ ことです。  
```java  
// Money クラス  
Class Money {  
  int 金額;  

  // コンストラクタ  
  public Money(int 入力){  
    this.金額 = 入力;  
  }  

  // ゲッター  
  public int get金額(){  
    return 金額;  
  }  
}  

// 用意するとき  
Money money = new Money(100)  

// 使うとき  
print(money.get金額())  
```  

こうすることで 必ず値の入ったオブジェクトを作ることができます。  

ただ コンストラクタの引数が int なので マイナスの値も入る問題があります。  
その対策として __コンストラクタの中で バリデーション(入力値チェック) をする__ と良いです。  

```java  
// Money クラス  
Class Money {  
  int 金額;  

  // コンストラクタ  
  public Money(int 入力){  
    if(入力 < 0){  
      throw Exception("マイナスはダメだよ")  
    }  
    this.金額 = 入力;  
  }  

  // ゲッター  
  public int get金額(){  
    return 金額;  
  }  
}  

// 用意するとき  
Money money = new Money(-1000) // エラーになる  

// 使うとき  
print(money.get金額())  
```  

これで このクラスは __常に安全な値の入った インスタンス__ を作ることができます。  
__コンストラクタで 確実に 正常値を設定しましょう__  

また バリデーション を 1か所にまとめる ことが できるので それぞれの部分での __エラーチェックが不要__ になります。  
例: 値をgetして それが null じゃないか みたいなチェック  

コンストラクタでバリデーションしないとき  
```java  
// あるファイルで...  
Money money = new Money()  
if(money.get金額() != null){  
  if(money.get金額() > 0){  
    print(money.get金額())  
  }  
}  

// 別のファイルで...  
Money money = new Money()  
if(money.get金額() != null){  
  if(money.get金額() > 0){  
    print(money.get金額())  
  }  
}  
```  
このように __重複したコード を 無くす__ ことができます。  

### ロジックは データを保持しているクラスに寄せる  
よくあるのは クラスは データを保持して 処理は 別のクラスに定義してあるメソッドで行う などです。  
```java  
Money money = new Money()  
計算.add(money, 10);  
```  

add() メソッドは Money クラス に関連する話なのに 計算 クラスに書かれてしまっています。  
この状態だと 両方のクラスが存在しないと 動作しません。  
また Money を修正したら 計算 のことも 気をつけないといけません。  

凝集度 を 高めるために Money クラス に定義しましょう  
__ロジックは データを保持しているクラスに寄せること__ が重要です。  

```java  
Class Money {  
  int 金額;  

  // コンストラクタ  
  public Money(int 入力){  
    if(入力 < 0){  
      throw Exception("マイナスはダメだよ")  
    }  
    this.金額 = 入力;  
  }  

  public add(int 足す額){  
    金額 = 金額 + 足す額;  
  }  
}  

// 使うとき  
Money money = new Money(100)  
money.add(10);  
```  

__関連するメソッドが 1個のクラスにまとまる__ ので __修正のときも__ このクラスを見ればよいと __分かりやすく__ なります。  
金額の修正してって言われたときに 計算クラスを探すより Money クラス を触る方が 直感的 です。  

### 変数を 不変にして バグを防ぐ  
インスタンス変数 の上書き は 理解が難しくなります。  
__値が 変わっていく前提__ だと いつ変更されたのか, 今の値がどうなっているのか __注意深く気をつける必要__ があります。  
また 仕様変更で処理が変わったとき 意図しない値に書き換わる かもしれません。  

```java  
// 商品金額  
int price = 950  
// 商品金額(税抜き) が 1000円以内なら 送料無料  
if(price < 1000){  
  // 送料無料 処理  
}  
// 税金計算  
price = 計算.税金(price)  
```  

仕様変更があって 税金計算するタイミングが変わりました  
本来は 送料無料なのに 無料じゃなくなります  

```java  
// 商品金額  
int price = 950  
// 税金計算  
price = 計算.税金(price) // この時点で 950*1.1 = 1045 になってしまった  
// 商品金額(税抜き) が 1000円以内なら 送料無料  
if(price < 1000){  
  // 送料無料 処理  
}  
```  

このようなバグを防ぐために __値が途中で書き換わらないよう 不変__ にしましょう  
- 変数に final を付けて 再代入できないようにする  
- セッターを 無くす  

### 変更したいときは インスタンスを作り直す  
値が不変になったら 変更するときどうするんだ! ってなると思います。  
そういうときは 変数を毎回 新たに定義したり 変更後の値を持った __インスタンス を 新たに作ればよい__ です。  

```java  
Class Money {  
  final int 金額; // 不変にする  

  // 良くない  
  public add(int 足す額){  
    // 再度代入できない  
    金額 = 金額 + 足す額;  
  }  

  // 良い  
  public Money add(int 足す額){  
    return new Money(金額 + 足す額);  
  }  
}  
```  

先ほどの 商品金額 でいうと 税込金額 を taxPrice とか 定義し直せばよい  

### 値の渡し間違い を 型 で防ぐ  
値の渡し間違いは 恐ろしいものです。  
以下の例だと 金額 に チケット枚数 を足し算できます。  

```java  
Class Money {  
  final int 金額;  

  // 良くない  
  public Money add(int 足す額){  
    return new Money(金額 + 足す額);  
  }  
}  

// 使うとき  
Money money = new Money(100)  
int ticketCount = 3 // 3枚  
// 引数が int なので 計算できてしまう!  
money01 = money.add(ticketCount);  
```  

__プログラマーの不注意__ から発生する可能性があります。  
こんなの 嫌ですよね?w  
どう 防げばよいでしょうか?  
それは __引数を Money 型 に 変更__ して 型が違うからダメ と __エディターに教えてもらえばよい__ のです。  

```java  
Class Money {  
  final int 金額;  

  // Money 型 にする  
  public Money add(Money 足す額){  
    return new Money(金額 + 足す額);  
  }  
}  
```  

こうすることで int は渡せなくなり バグを減らすことができます。  
なるべく プリミティブ型(int, String, Boolean など) は 使わず __独自の型 を 作っていきましょう__。  
また 独自の型 を 作っていくことで __コードが ドキュメント化する__ ので とても読みやすくなります。  
`add(int 変数)` だと int のなにか ということしか読み取れません。  
`add(Money 変数)` だと 金額 を渡せばいいのね! と 意図が伝わりやすくなります。  

## どんな クラス単体で動作する の まとめ  
上記のように 自己防衛責務 を持たせて 凝集度 を高めると 以下のようなメリットがあります。  
- コンストラクタ で インスタンス変数 の 値 が 確定するため __初期化してない状態を防げる__  
- コンストラクタ で バリデーション するため __不正な インスタンス が作られない__  
- 必要なロジックが 1個のクラスに集まるため あちこちに __ロジックが分散せずに 修正が容易になり, 修正漏れも減る__  
- 必要なロジックが 1個のクラスに集まるため __重複したコードが減る__  
- 変数など 不変にすることで __思わぬバグを防ぐ__  
- 独自の型 を 引数 に使うことで __間違った値を 渡さなくなる__  

## まとめ  
クラス設計 とは __インスタンス変数 を 不正な状態 に 陥らせないための 仕組みづくり__ です。  
密接に関係する __ロジック が 1か所 に集まっている__ ため __凝集度が高い__ です。  
つまり 保守性の高いコード になりました。  
このように データと操作するロジックを1つのクラスにまとめ 必要な処理(メソッド)のみを外部に公開することを __カプセル化__ といいます。  

この考え方をベースに オブジェクト指向とは, ドメイン層のオブジェクトとは など より勉強していきましょう!
