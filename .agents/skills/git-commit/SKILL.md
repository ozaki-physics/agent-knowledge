  コミット作業を段階的に進めるスキル。差分を確認 → 粒度を提案 → メッセージ案を提示 → 承認を得てから実行、という安全なフローで git commit を行う。
  「コミットして」「差分をコミット」「コミット粒度を教えて」「どんなコミットがよい」「変更をコミット」「次の差分ならどういうコミットがよい」などの指示があったときに必ず使う。
  コミット関連の作業であれば、ユーザーが明示的にスキル名を言わなくても積極的に使う。
---

## フロー

### 1. 差分を確認する

まず現在の状態を把握する。

```bash
git status
git diff HEAD
```

ファイル一覧だけでは判断できない場合は、利用可能なファイル読み取り手段で各ファイルの内容を確認する。

### 2. コミット粒度を提案する

以下の基準でグループを決める。

- **変更の理由が1つにまとまるか** を基準にする
- 性質が異なるファイル (設定 / ドキュメント / ソースコード / メモ) は分ける
- 関連性が高いファイルは1コミットにまとめる

提案はテーブル形式で示す。

| # | コミットメッセージ (案) | 対象ファイル |
|---|---|---|
| 1 | `feat: ...` | `src/foo.ts` |
| 2 | `docs: ...` | `README.md` |

### 3. ユーザーの確認を取る

提案を示した後、必ず確認を取る。ユーザーが「2コミットに分けたい」「まとめてほしい」などの調整を要求した場合は柔軟に対応し、再度メッセージ案を提示する。

**承認が得られるまで git commit は実行しない。**

承認の例: 「お願いします」「コミットして」「進めて」

### 4. コミットを実行する

承認が得られたら、提案した順番通りに1つずつ実行する。

```bash
git add <対象ファイル>
git commit -m "<メッセージ>"
```

全コミット完了後、`git status` で未コミットの差分が残っていないか確認する。

## コミットメッセージのルール

- **言語**: 日本語
- **プレフィックス**: `feat / fix / docs / refactor / test / chore` のいずれか
- **形式**: `プレフィックス: 説明`

| プレフィックス | 使いどころ |
|---|---|
| `feat` | 新しい機能・ファイルの追加 |
| `fix` | バグ修正 |
| `docs` | ドキュメント・メモ類 |
| `refactor` | 動作を変えないコード整理 |
| `test` | テストの追加・修正 |
| `chore` | 設定ファイル・ツール類 |

**例:**
- `feat: オザキスタイルで文章を書くスキル を 追加`
- `docs: 社内勉強会 の 登壇資料 を 追加`
- `chore: VSCode の 設定 を 追加`
