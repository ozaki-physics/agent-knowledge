# Design Document の目次

## 運用
- 運用ドキュメント は `docs/design-docs/` 配下で管理する
- ファイル分割は "読みやすさ" と "変更しやすさ" を優先して決める
- 小さな機能は 1ファイルにまとめてよい
- 大きな機能や長期運用の対象は `概要` と `詳細` (API / DB / フロー / エラー設計 など) に分割してよい
- 設計ドキュメント は `service` と `feature` が分かる形で管理する
- ファイル名は `NNNN-<service>-<feature>.md` のようにしてよい
- 新しい設計ドキュメントを追加したら この目次に追記する

## 書き方
- 最低限 `何をどう実装するか` が分かるように書く
- まず 結論(採用案)を短く書く
- `service` と `feature` は先頭で明示する
- DB, バックエンド, フロントエンド の内容は 必要なら同じファイルの中で章を分けて書く
- 1ファイルが 長くなったら分割してもよいが 必須ではない
- 新規作成時は `docs/design-docs/0000-template.md` をコピーして使う

## 命名ルール
- `NNNN` は 4桁連番とする
- `service` は `info`, `capital` のようにサービスを識別できる名前にする
- `feature` は `auth-n` のように機能を識別できる名前にする
- ファイル名は kebab-case を使う
- 同じ `feature` 名でも `service` が違えば別ドキュメントとして扱う
- 同じ機能に関する設計は 基本は 1ファイルにまとめる
- 1ファイルが大きくなってきたら バックエンド, フロントエンド, DB, Overview など ファイル分割を考える

## ADR との違い
- design-docs は "どう実装するか" を記録する
- ADR(`docs/adr`) は "なぜ その方針を選んだか" を記録する
- 方針決定が必要なテーマは ADR を先に作成して 実装詳細を design-docs に落とし込む
