# ADR 0001: フォルダ名の単語区切りは kebab-case を原則とする
ステータス: Accepted

## 背景(コンテキスト)
このリポジトリは エージェント運用, コーディング規約, ADR, design-docs などの知識や方針を管理する。
そのため フォルダ名は コード上の識別子として扱うよりも, 人間が読む文書パスや GitHub 上の表示として扱う場面が多い。

一方で 将来的に サンプルコード, スクリプト, import されるモジュール を置く可能性もある。
その場合は フォルダ名が 言語や実行環境の制約を受ける。

## 課題
フォルダ名で 単語区切りに ハイフン(`-`) と アンダースコア(`_`) のどちらを使うかを統一し, 次の問題を避ける。

- 同じ意味のフォルダ名が `domain-model` と `domain_model` のように混在する
- ドキュメントパスとして読みにくい名前になる
- import 対象のフォルダに ハイフンを使い, 言語上扱いにくくなる
- 個人ごとの好みで 命名が揺れる

## 決定
このリポジトリのフォルダ名は 原則として `kebab-case` を使用する。

例:

```text
coding-rules/
git-rule/
domain-driven-design/
review-guidelines/
use-case-layer/
```

ただし, プログラムから import されるモジュールやパッケージ配下では 対象言語の慣例を優先する。

例:

```text
# Python の import 対象
domain_model/
use_case/

# ドキュメント用フォルダ
domain-model/
use-case/
```

また, 原則として 次の命名は避ける。

- 大文字小文字の混在
- 空白を含むフォルダ名
- 日本語フォルダ名
- 同じ階層での `kebab-case` と `snake_case` の混在

## 検討した選択肢
1. フォルダ名を `kebab-case` に統一する
2. フォルダ名を `snake_case` に統一する
3. 用途によって `kebab-case` と `snake_case` を使い分ける

## 決定理由
このリポジトリでは ドキュメント, 方針, 設定, ADR のように 人間が読む情報を管理する比率が高い。
そのため URL, Markdown, GitHub 上の表示, 文書パスとして読みやすい `kebab-case` を原則にする。

`kebab-case` は Web, URL, GitHub リポジトリ名, npm パッケージ名, ドキュメントサイト でよく使われる。
英単語の区切りとして自然に読めるため, `domain-driven-design` や `review-guidelines` のような名前と相性がよい。

一方で `kebab-case` は コード上の識別子としては扱いにくい。
たとえば Python では `domain-model` をそのまま import できない。
そのため import 対象になるフォルダでは `snake_case` など 対象言語の慣例を優先する。

この判断により, ドキュメントとしての読みやすさを優先しつつ, 実行コードやモジュールの扱いやすさも保つ。

## メリット: 期待される効果
- ドキュメントパスや GitHub 上の表示が読みやすくなる
- リポジトリ名 `agent-knowledge` と命名方針を揃えられる
- URL や Markdown リンクと相性がよい
- フォルダ命名の迷いを減らせる
- import 対象では 言語慣例を優先できるため 実装上の制約にも対応できる

## デメリット: 受け入れるリスク
- Python などの import 対象フォルダでは `kebab-case` を使えないため 例外ルールが必要になる
- `kebab-case` と `snake_case` の使い分け判断が必要になる
- 既存フォルダを後から改名する場合, Markdown リンクや参照パスの修正が必要になる

## 影響範囲
- コード: import 対象のフォルダでは 対象言語の慣例を優先する
- テスト: フォルダ名変更のみの場合は 原則として追加テスト不要。ただし import パスや参照パスを変更した場合は 関連テストを実行する
- 運用: 新しいフォルダを作成するときは 原則 `kebab-case` を使う。例外が必要な場合は 理由を README, ADR, または近い場所のドキュメントに残す

## 将来の考慮事項
- 実行コードを置くディレクトリが増えた場合は, 言語ごとの命名規則を別 ADR または コーディング規約に分離する
- `docs/`, `src/`, `scripts/` など トップレベルごとの命名方針が必要になった場合は, この ADR を置き換える
- 既存フォルダをリネームする場合は, Markdown リンク, CI 設定, スクリプト, README の参照を合わせて更新する

## 参考
- このリポジトリでは ドキュメント用フォルダを `kebab-case`, import 対象を 言語慣例 として扱う
