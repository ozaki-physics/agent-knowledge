#  テックブログスタイル ライター

テックブログシリーズと同じテイストで技術ドキュメントを作成するためのガイドです。
著者 の文体, 構成を忠実に再現することが目的です。

## スタンス

このドキュメントシリーズのキーワードは **「親しみやすさ」と「段階的な論理展開」** です。

- 一人称は「僕」を使う
- 読者への問いかけで導入する（「〜したことはありませんか?」「〜ですよね?」）
- 技術を **業務, 保守性, 価値提供** の文脈で位置づける（業務の具体例: 受注, 商品, 在庫 など）
- まとめは命令形で締める（「〇〇しなさい」）
- 難しい概念は「ふわっとした定義」→「詳細」の2段階で説明する

---

## ドキュメント構造（必須）

以下の構造を守ること。セクションの順番も重要。

```
# [タイトル] について  
## 導入  
## 目次  
## 目的  
## [メインセクション群（H2/H3/H4）]  
## まとめ  
## その他 参考文献  

<style>...</style>   ← 必ずファイル末尾に追加
```

### 行末スペース

各行末に半角スペース2つを入れる（Markdownの改行を正確に制御するため）。
これはこのシリーズ全体の書式習慣なので忠実に従う。

---

## 目次セクション

アンカーリンク付きの階層型目次を必ず入れる。
見出しの日本語テキストはそのままリンクに使い、スペースを `-` に変換する。

```markdown
## 目次  
- [目的](#目的)  
- [セクション名](#セクション名)  
  - [サブセクション名](#サブセクション名)  
- [まとめ](#まとめ)  
- [その他 参考文献](#その他-参考文献)  
```

---

## 導入セクション

身近な気づきなどから始めて、読者の興味を引く。
一般的な内容の話題から自然につなげる。
一般的な話題例: クラス設計, 凝集度/結合度, 保守性, クリーンアーキテクチャ, TDD, DDD など。

問いかけから始めて「なぜこのトピックが必要か」を会話的に導く。

```markdown
## 導入  
前回まで 〇〇 について 話してきました。  
〇〇で 保守性の高いコードが書けるようになりました。  

では 〇〇の場合は どうでしょうか?  
「〜したことはありませんか?」  

今回は __〇〇__ について 話していきます。  
```

---

## 目的セクション

箇条書きなし。1行1目的で書く。動詞で終わらせる。

```markdown
## 目的  
〇〇を学ぶ  
〇〇を理解する  
〇〇の判断基準を得る  
```

---

## 文体ルール

### 強調

重要ポイントは `__太字__`（アンダースコア2つ）で囲む。
**Markdown の `**太字**` ではなく `__太字__` を使う。**

### 技術概念の説明パターン

まずざっくり説明し、その後に詳細を展開する。

```markdown
### 〇〇とは  
ざっくり言うなら __〇〇のこと__ です。  

正確には 〇〇は 〇〇であり...  
```

### 折りたたみ要素

詳細な例や補足情報は `<details><summary>` で折りたたむ。
必ず `class="details-content-indent"` を付けた Div で中身を囲む。

```html
<details>  
    <summary>例のタイトル</summary>  
    <div class="details-content-indent">  
      <div>悪い例: ...</div>  
      <br>  
      <div>良い例: ...</div>  
    </div>  
</details>  
```

### コード例

VB.NET / Java / C# を使う。必ず **悪い例 → 良い例** のペアで示す。
コードブロック(` ``` `)で書く場合は言語指定を添える。

```markdown
```java  
// 悪い例のコード  
```  

```java  
// 良い例のコード  
```  
```

また、折りたたみの中でコードを示すときは `<div>` で1行ずつ書く方法も使う（既存ファイル参照）。

### テーブル

比較や指標の表は Markdown テーブルで書く。ヘッダーは短く。

```markdown
| 指標 | 意味 |
| - | - |
| 〇〇 | 〇〇 |
```

---

## まとめセクション

箇条書きで、重要な点を `__太字__` で強調。
最後の1〜2行は必ず命令形で締める。

```markdown
## まとめ  
- __〇〇とは 〇〇のこと__  
- __〇〇が重要な理由は 〇〇__  
- 〇〇を意識して __〇〇しなさい__  
```

---

## 参考文献セクション

外部リンクは必ず `<a>` タグでセキュアに書く。書籍は著者名を添える。

```markdown
## その他 参考文献  
- <a href="URL" target="_blank" rel="noopener noreferrer">書籍タイトル</a> (著: 著者名)  
- <a href="URL" target="_blank" rel="noopener noreferrer">記事タイトル</a>  
```

---

## ファイル末尾のCSSブロック（必須）

**全ドキュメントの末尾に必ず追加する。** `assets/common_style.css.md` の内容をそのまま貼り付ける。
1文字も変えない。

## 業務・保守性の文脈

このドキュメントシリーズは業務システムの内部品質向上を目的とした教育資料。
どんな技術トピックでも以下の観点を必ず入れる。

1. **なぜ保守性が上がるのか** を説明するセクションを作る
2. 以下の大前提と接続させる
   - 「コードは必ず変更される」
   - 「コードは書くために使った時間より読まれる時間の方が長い」
3. 業務の言葉で具体例を作る（受注, 商品, 在庫, 金額計算 など）
4. 「変更の影響範囲を小さくする = 保守性が高い」という論理を活用する
